# llm.txt — AI Agent Brief

A dense brief for AI coding agents working in this repo. Read this first. For full detail, see `CONVENTIONS.md` (coding standards) and `README.md` (quick start).

## First Step: Detach From the Template Repo

**Before you touch anything else, run `bash detach.sh`.** The template is cloned from `erium/app-template`; leaving that remote attached risks accidentally pushing an app's work back upstream. The script is idempotent:

- If `origin` points at `erium/app-template`, it wipes `.git/` and runs `git init` + initial commit so your new project has a clean history.
- If there is no `.git/`, it initializes one.
- If `origin` is anything else (or missing), it exits without changes.

After detaching, add your own remote with `git remote add origin <your-repo-url>`. Do not push before running this script.

## What This Is

A generic full-stack TypeScript template: React frontend + Next.js Route Handlers + PostgreSQL via Drizzle ORM. Ships with auth, multi-tenancy (tenant → users → invitations), i18n (de/en), Stripe billing, PDF export (Playwright), and 53 shadcn/ui components. Single `package.json` (no monorepo). Branding lives in `shared/config.ts`.

## Silent-Failure Gotchas — Memorize These

These rules are silent: violating them breaks the app without an obvious error message. Check this list before writing any new file. **When something fails: read logs first, code last** — the fix is almost always visible in the error output. Do not guess.

- **`create_app` MUST use `bash start.sh --dev <PORT>` (debug) or `bash start.sh <PORT>` (prod) — nothing else.** The generic `create_app` template in your system prompt suggests `pnpm dev` / `pnpm install` / `bash setup-postgres.sh` / `pnpm db:push` chains — DO NOT use them here. `start.sh` already runs all of that plus Node 20 upgrade and `NEXT_PUBLIC_BASE_PATH` setup. Skipping it = wrong Node version + Halerium sub-path broken + assets 404.
- **TWO apps per project, always: `<app>_debug` AND `<app>` (production).** Single-app setup is forbidden — production verification has no home, and you'll forget to add it later. Create both up front, save both friendly URLs in the plan, develop on debug, verify on prod when a feature is done.
- **Run `pnpm check` (and `pnpm lint`) BEFORE `start_app` / `restart_app`.** Next.js compile errors crash the dev server on boot but DO NOT reach pino — `app-startup.log` and `logs/error.*.log.ndjson` stay clean while the app crash-loops. If you skipped the type-check, you'll waste hours staring at empty logs. Crash-loop with clean logs = run `pnpm check` and `pnpm build` locally; the failing line is in their output.
- **Every file in `src/views/` MUST start with `"use client"` on line 1.** Without it, hooks like `useState`/`useQuery` throw a server-component error that's hard to trace.
- **Guard all `localStorage` / `window` / `document` access with `if (typeof window === "undefined") return;`** — Next.js SSRs `"use client"` components for the initial HTML, so top-level access crashes the server render.
- **i18n keys must exist in BOTH `src/locales/de/common.json` AND `src/locales/en/common.json`.** Missing one falls back silently to the key string — no error, just garbled UI.
- **All drizzle queries go in `server/db.ts`.** Don't write `db.select()` in a route handler; route handlers call helpers from db.ts.
- **Server code uses `logger` from `server/utils/logger.ts`, never `console.*`.** Console output bypasses NDJSON log files and is invisible to debugging.
- **Browser must navigate the friendly URL from `create_app`, never `localhost:3000`.** Halerium serves at a sub-path; localhost loads but assets 404.
- **Schema changes go through `pnpm db:generate` → review SQL → `pnpm db:migrate`.** Never `db:push` after seeding real data; never run raw `CREATE TABLE` / `ALTER TABLE`.

## Commands

```bash
pnpm check        # tsc --noEmit — run after every edit
pnpm lint         # eslint .   — run after every edit (use lint:fix for autofixes)
pnpm build        # next build → .next/ — run once at end of each feature
pnpm db:generate  # generate a migration from schema.ts changes
pnpm db:migrate   # apply pending migrations (use this, not db:push)
pnpm db:push      # destructive schema sync — first-time local setup ONLY
pnpm db:seed      # idempotent seed: tenant + admin@example.com / ChangeMe!2026
pnpm dev          # next dev — local-only; agents use start.sh --dev via Control App
pnpm start        # next start — local-only; agents use start.sh via Control App
```

PostgreSQL setup: `bash setup-postgres.sh` (apt-installs locally, not Docker). Uninstall: `bash setup-postgres.sh uninstall`.

## Build & Run

The app runs via the Control App capability with a two-app pattern (debug + production) — see **setup.md §3** for the workflow. `start.sh` handles the full bootstrap on every spin-up: Node 20, pnpm, Postgres install + start, schema push, seed, build (prod only), and run. Do not invoke `pnpm dev` / `pnpm start` / `setup-postgres.sh` / `pnpm db:push` / `pnpm db:seed` directly — the Control App owns the lifecycle.

The dev-loop scripts you DO run yourself are `pnpm check`, `pnpm lint`, and `pnpm build` — see **setup.md §4** for the inner-loop sequence (edit → check → lint → HMR → browser → log → build).

The friendly URL returned by `create_app` is what the browser navigates to (never `localhost:3000`). Save both URLs in your plan.

### Log Files

Logs live in `logs/` at the repo root. Format: **NDJSON** (one JSON object per line, ISO 8601 timestamps). One file per day per stream — restarts on the same day append to the existing file. See **setup.md §5** for the find-newest-by-mtime recipe.

| Log | Path | Purpose |
|---|---|---|
| App log | `logs/app.<YYYY-MM-DD>.log.ndjson` | All levels (debug+ in dev, info+ in prod). |
| Error log | `logs/error.<YYYY-MM-DD>.log.ndjson` | Errors only — check first when something breaks. |
| Startup log | `app-startup.log` (repo root) | `start.sh` bootstrap output. Read first if the app fails to start. |
| PostgreSQL | `pg-data/pg.log` | Database daemon log. Check on `ECONNREFUSED 127.0.0.1:5432`. |

### Common Failures

| Symptom | Cause | Fix |
|---|---|---|
| Blank page or 404 on all routes | `.next/` missing or stale — Next.js production build not run. | Run `pnpm build`, check for errors. |
| `Port XXXX is busy` in logs | Another process holds the port. | Server auto-finds the next free port. Check log output for the actual port. |
| Database connection refused | PostgreSQL not running. | `bash setup-postgres.sh` (local) or `bash start.sh` (runner). Check `pg-data/pg.log`. |
| API returns 401 on every request | `JWT_SECRET` not set. | Add `JWT_SECRET` to `.env`. Generate with `openssl rand -hex 32`. |
| `next: command not found` | Dependencies not installed. | `pnpm install` first. |

## Database Migrations

Schema changes **must** go through Drizzle migrations so that every change is versioned, reviewable, and reproducible. Never alter the database directly with raw SQL.

### Workflow

```bash
# 1. Edit the schema definition
#    → drizzle/schema.ts

# 2. Generate a migration (creates a numbered .sql file)
pnpm db:generate

# 3. Review the generated SQL in drizzle/migrations/NNNN_*.sql

# 4. Apply the migration
pnpm db:migrate

# 5. Commit the migration file — it is version-controlled
git add drizzle/migrations/
```

### Commands

| Command | What it does | When to use |
|---|---|---|
| `pnpm db:generate` | Diffs `schema.ts` against the last migration snapshot, writes a new `.sql` file. | After every schema change. |
| `pnpm db:migrate` | Applies all pending `.sql` migrations in order. | Dev, CI, production — always. `start.sh` runs this automatically. |
| `pnpm db:push` | Destructively syncs `schema.ts` to the database without migration files. | **First-time local setup only**, before any real data exists. Never in production. |
| `pnpm db:seed` | Idempotent seed: default tenant + admin user. | After migrations, on first setup. |

### Rules

- **Always commit migration files.** The `.sql` files in `drizzle/migrations/` and the `meta/` journal are the authoritative record of every schema change. Without them, `db:migrate` on another machine or runner has nothing to apply.
- **Never run raw SQL** (`CREATE TABLE`, `ALTER TABLE`, `DROP`) to change the schema. All changes go through `drizzle/schema.ts` → `pnpm db:generate`.
- **Never use `db:push` in production** or on any database with real data. It can drop columns and lose data without warning.
- **Review generated SQL before applying.** `db:generate` is not infallible — check that the `.sql` makes sense (e.g., no unexpected `DROP COLUMN`).

## Request Lifecycle

```
Browser ─fetch→ Next.js Route Handler (app/api/*/route.ts)
 ─→ try { requireUser(request) }   reads app_session_id cookie → verifies JWT → throws Response(401) if missing
 ─→ zod.parse(body | params)
 ─→ server/db.ts helper ─→ drizzle-orm ─→ pg pool ─→ PostgreSQL
 ← NextResponse.json({ data }) or NextResponse.json({ error }, { status: 4xx })
```

## Auth Tiers

Three levels, enforced per-handler via helpers from `src/server/getUser.ts`:

| Tier | Helper | Examples |
|---|---|---|
| Public | (none) | `POST /api/auth/login`, `POST /api/auth/register`, `GET /api/health` |
| Protected | `requireUser(request)` | `GET /api/auth/me`, `POST /api/payment/checkout` |
| Admin | `requireAdmin(request)` | `POST /api/tenant/invite`, `DELETE /api/tenant` |

Both helpers throw a `Response` object on failure. Wrap the handler body:

```ts
export async function GET(request: Request) {
  try {
    const user = await requireUser(request);   // throws Response(401) if unauthenticated
    // const user = await requireAdmin(request); // throws Response(403) if not admin
    // ... handler logic ...
    return NextResponse.json({ ... });
  } catch (err) {
    if (err instanceof Response) return err;
    logger.error({ err }, "[Resource] handler error");
    return NextResponse.json({ error: "Internal server error" }, { status: 500 });
  }
}
```

## Feature Recipe

To add a new feature end-to-end:

1. **Schema (if needed)** — edit `drizzle/schema.ts`, run `pnpm db:generate` to create a migration, review the SQL, then `pnpm db:migrate` to apply it. Commit the migration file.
2. **DB helpers** — add async functions to `server/db.ts` using drizzle-orm. Use `.returning()` for writes, destructure `[row]` for single-row reads.
3. **API route** — create `app/api/<resource>/route.ts` exporting `GET`/`POST`/`PUT`/`DELETE` functions. Call `requireUser` or `requireAdmin` from `src/server/getUser.ts`. Validate body with zod. Return `NextResponse.json(...)`.
4. **API client method** — add a typed entry to `src/lib/api.ts`. Add its query key to `queryKeys` if used with react-query.
5. **Page or component** — create a `"use client"` component in `src/views/` or `src/components/`. Fetch via `useQuery({ queryFn: api.<method> })`. If it's a page, also create `app/<route>/page.tsx` that re-exports it: `export { default } from "@/views/MyPage"`.
6. **Sidebar entry (if it's a page users navigate to)** — add an item to `menuGroups` in `src/components/DashboardLayout.tsx`: `{ icon, label: "<i18n_key>", path: "/your-page", roles?: ["admin"] }`. Skip this step if the page is intentionally opt-in by URL only.
7. **i18n** — add every user-facing string as a key in `src/locales/de/common.json` AND `en/common.json`. Use `const { t } = useTranslation(); t("key")`. The sidebar `label` is an i18n key, not a literal string.
8. **Verify** — `pnpm check`, `pnpm lint`, browser test at the debug-app friendly URL (HMR has reloaded already), then `pnpm build` once when the feature is fully done.

## Quick Patterns

Drizzle single-row query:
```ts
const [user] = await db.select().from(users).where(eq(users.id, id)).limit(1);
```

Drizzle insert with returning:
```ts
const [row] = await db.insert(users).values(data).returning();
```

Drizzle transaction:
```ts
await db.transaction(async (tx) => {
  await tx.insert(tenants).values(...);
  await tx.insert(users).values(...);
});
```

Client query:
```ts
const { data } = useQuery({ queryKey: queryKeys.me, queryFn: api.getMe });
```

Client mutation:
```ts
const mutation = useMutation({
  mutationFn: api.login,
  onSuccess: () => toast.success(t("login_success")),
});
```

Next.js Route Handler with validation:
```ts
// app/api/auth/login/route.ts
const schema = z.object({ email: z.string().email(), password: z.string().min(6) });
export async function POST(request: Request) {
  try {
    const body = schema.parse(await request.json());
    // ...
    return NextResponse.json({ success: true });
  } catch (err) {
    if (err instanceof Response) return err;
    return NextResponse.json({ error: "Internal server error" }, { status: 500 });
  }
}
```

## Deployment (sub-path / Halerium)

The template is deployable behind a reverse-proxy sub-path (e.g. Halerium mounts apps at `/apps/<org>/<ws>/<AppName>/`). Two layers are already wired — do not duplicate this logic:

- **Next.js assets + routing** — `next.config.ts` sets `basePath: process.env.NEXT_PUBLIC_BASE_PATH ?? ""`. All `<Link>` hrefs, `router.push()`, and asset URLs automatically prepend the base path.
- **API client** — `src/lib/api.ts` prepends `BASE_PATH` via `getApiBase()` from `src/lib/basePath.ts`.

`NEXT_PUBLIC_BASE_PATH` is a **build-time** constant. `start.sh` exports it before `pnpm build` when `HALERIUM_ID` is set. If the runner is recycled with a new `HALERIUM_ID`, delete `.next/` to force a rebuild.

**Runner bootstrap:** `start.sh` (at repo root) idempotently upgrades Node to v20, installs pnpm + Postgres, handles stale `postmaster.pid` after ephemeral spin-downs, pushes schema, seeds, and launches via package.json scripts only:

- `bash start.sh [PORT]` — prod: builds if needed (`if [ ! -f .next/BUILD_ID ]`), runs `pnpm start`.
- `bash start.sh --dev [PORT]` — dev: runs `pnpm dev` (Next.js HMR). Use this while iterating.

Use runner type **`nano`** on Halerium (`standard` silently fails).

Full detail in `DEPLOYMENT.md`.

## How to Add AI Chat to Your App

The template ships with a Vercel AI SDK example at `/chat-example` (requires login, linked from the sidebar as "AI Chat"). Use it as the starting point for any LLM-powered feature.

**Env vars** (add to `.env` — all optional; endpoint returns 503 without a key):

```
OPENAI_API_KEY=sk-...
OPENAI_BASE_URL=https://api.openai.com/v1   # override for OpenAI-compatible endpoints
OPENAI_MODEL=gpt-4o-mini
```

`OPENAI_BASE_URL` lets you swap in Azure OpenAI, OpenRouter, vLLM, or a local Ollama (via its OpenAI-compat shim) without code changes.

**Server — `app/api/chat/route.ts`** — calls `requireUser`, zod-validates `{ messages }`, returns `streamText(...).toDataStreamResponse()`. Swap providers by installing `@ai-sdk/anthropic` (or similar) and replacing `openai(model)` with `anthropic(model)`.

**Client — `src/components/chat/ChatExample.tsx`** — uses `useChat` from `@ai-sdk/react` with `api: \`${getApiBase()}/chat\`` and `credentials: "include"`. The `getApiBase()` call is required — hard-coding `/api/chat` breaks sub-path deployments.

**To add a new AI feature**: copy `app/api/chat/route.ts`, adjust the system prompt / messages shape, create a `src/views/` page that uses `useChat` (chat) or `useCompletion` (single-turn) pointed at the new endpoint.

## Anti-Patterns

Never:

- Deploy without building first. `pnpm start` serves from `.next/` — run `pnpm build` and check its output for errors before starting. On runners, use `start.sh` (it builds automatically).
- Claim the app is working without verifying. After `start_app`, open the debug-app friendly URL in the browser, exercise the feature, and read the newest `logs/error.*.log.ndjson`. If something fails, read the logs (see **Log Files** above) before making more changes.
- Run `db:push` against a production database or any database with real data. Use `pnpm db:generate` + `pnpm db:migrate` instead. See **Database Migrations** above.
- Skip committing migration files. The `.sql` files in `drizzle/migrations/` are version-controlled — they are the authoritative record of every schema change.
- Run raw SQL (`CREATE TABLE`, `ALTER TABLE`, `DROP`) to change the schema. All changes go through `drizzle/schema.ts` → `pnpm db:generate`.
- Reintroduce tRPC or Express. The API uses Next.js Route Handlers by design.
- Add `axios`. Use the `fetch`-based `api.*` wrapper.
- Call `fetch` directly from a component. Go through `src/lib/api.ts`.
- Scatter drizzle queries across route handlers. Put them in `server/db.ts`.
- Hardcode `APP_NAME`, support email, or copy strings. Use `shared/config.ts` and i18n keys.
- Assume the app is served from `/`. Use `BASE_PATH` / `getApiBase()` from `src/lib/basePath.ts` for client-side URL construction.
- Use `console.log/error/warn` in server code. Import `logger` from `server/utils/logger.ts` and use `logger.info/warn/error/debug`. Pass errors as `logger.error({ err }, "message")`. Logs land in `logs/app.<date>.log.ndjson` (all levels) and `logs/error.<date>.log.ndjson` (errors); one file per day per stream — restarts on the same day append. Dev defaults to `debug`, prod to `info`; override with `LOG_LEVEL`.
- Access `localStorage` or `window` in component top-level state initializers — Next.js SSRs `"use client"` components for the initial HTML. Guard with `if (typeof window === "undefined")`.
- Commit `.env`, `pg-data/`, `uploads/`, `data/`, `.next/`, `logs/`.
- Skip git hooks with `--no-verify`.
- Use `any`, `console.log` in committed code, or browser `alert`.
